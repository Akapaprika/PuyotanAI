# PuyotanAI スケールアップ計画（最小エンジン → 本格実装）

## 0. 現状（最小エンジン）
- 盤面 `GameState` があり、Python から `choose_move` を呼べる。 
- ただし、ゲームロジック（落下・連鎖・評価・探索・学習）はまだ薄い。

---

## 1. 本格化までの全体フロー（6フェーズ）

### フェーズ1: **ルールエンジンの完成（正確性の土台）**
1. 盤面更新 API を定義（ぷよ設置、重力、消去、連鎖）。
2. 連鎖シミュレーションを実装（1手適用→連鎖結果まで）。
3. 「合法手列挙」を実装（回転・列制約込み）。
4. 入出力の安定化（C++ と Python のデータ構造を固定）。
5. 単体テストでルールを固定化（後方互換の守り）。

**完了条件**
- 同じ入力盤面 + 手で、常に同じ結果（決定性）が出る。
- サンプル局面で期待連鎖数・消去数がテストで一致する。

### フェーズ2: **評価関数 v1（強くする入口）**
1. ヒューリスティクス特徴量を設計（高さ、穴、連結、発火可能性など）。
2. スコア式を実装し、重みを設定（初期は手調整）。
3. ランダム対戦で勝率をベースライン化。

### フェーズ3: **探索器 v1（実戦強度）**
1. 先読み深さ固定のビームサーチ or ミニマックスを実装。
2. 時間制限 API（例: 50ms/手）を追加。
3. 探索ログ（展開ノード、深さ、最終評価）を可視化。

### フェーズ4: **学習ループ（性能改善の自動化）**
1. 対戦ログ保存フォーマットを決定（JSONL 推奨）。
2. 自己対戦ジョブを回し、データ収集。
3. 重み最適化（進化戦略や回帰）で評価関数を更新。

### フェーズ5: **実運用基盤（大規模化対応）**
1. 再現可能実験（seed、config、モデル版管理）。
2. CI でテスト + ベンチ（速度劣化検知）。
3. 指標ダッシュボード（勝率、平均探索ノード、クラッシュ率）。

### フェーズ6: **高度化（上位強度）**
1. NN 評価器への置換（特徴量手設計から移行）。
2. MCTS / ハイブリッド探索。
3. 相手モデル推定・局面タイプ別戦略。

---

## 2. 「第一歩」でやること（今週の実装タスク）

最初に着手すべきは **フェーズ1の「ルールエンジン完成」**。

### タスクA: C++ 側に「1手適用 API」を作る
- 追加 API:
  - `apply_move(GameState&, Move)`
  - `resolve_chain(GameState&) -> ChainResult`
- `ChainResult` に最低限入れる情報:
  - `chains`（連鎖数）
  - `erased`（総消去数）
  - `score`（暫定スコア）

### タスクB: Python から検証しやすい境界を作る
- pybind11 で `ChainResult` を公開。
- Python 側で `simulate_once(state, move)` を呼べるようにする。

### タスクC: 最低5ケースの確定テストを作る
- 例:
  1. 落下のみ（消去なし）
  2. 1連鎖
  3. 2連鎖
  4. 列あふれ（不正手）
  5. 同形盤面の再現性（同じ結果が出る）

## 4. 進捗管理の最小 KPI
- ルールテスト成功率: **100%**
- 1手シミュレーション速度: **目標 1ms 以下（ローカル）**
- ランダム対戦 100 試合のクラッシュ率: **0%**

---

## 5. まずこの順で着手する理由
1. ルールが不安定だと、探索・学習の評価が壊れる。
2. 先に API とテストを固定すると、後続の改善で壊れにくい。
3. 最小エンジンから本格化する最短経路は「正確なシミュレータ化」。
